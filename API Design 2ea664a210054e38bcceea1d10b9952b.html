1. API? main purpose?
  • API（应用程序编程接口）是允许不同软件应用程序相互通信的规则和协议集合，其主要目的是集成、抽象、重用、标准化、提高效率、可扩展性、安全性和灵活性。

    An API (Application Programming Interface) is a set of rules and protocols that allows different software applications to communicate with each other, primarily aiming for integration, abstraction, reusability, standardization, efficiency, scalability, security, and flexibility.

2. Differences between API and Wed services
  • all web services are APIs, not all APIs are web services. APIs have a broader scope and can be used in various contexts, while web services are specifically designed for network-based communication基于网络的通信而设计, often over the web.

    FlexibilityHighModerate

3. RESTful API Principles?

  • 1. Stateless（无状态）

    场景：
    想象你在开发一个分布式系统。每个请求都是独立的，没有依赖于以前的请求状态。你可以自由地在不同服务器上处理每个请求，这样系统的扩展性和可靠性就大大提高了。

    记忆点：
    每个请求都是一个完整的“快照”，无需记住之前的状态。

    Scenario:
    Imagine you are developing a distributed system. Each request is independent and does not rely on previous request states. You can freely handle each request on different servers, greatly improving the scalability and reliability of the system.

    Memory Tip:
    Each request is a complete "snapshot" with no need to remember previous states.

    2. Client-Server（客户端-服务器）

    场景：
    想象你在构建一个移动应用程序（客户端）和一个后台服务（服务器）。客户端负责用户界面和体验，而服务器处理数据存储和业务逻辑。两者通过API进行通信，各自独立开发和扩展。

    记忆点：
    客户端和服务器分离，职责清晰。

    Scenario:
    Imagine you are building a mobile application (client) and a backend service (server). The client handles the user interface and experience, while the server manages data storage and business logic. They communicate through the API, allowing for independent development and scaling.

    Memory Tip:
    Client and server are separated with clear responsibilities.

    3. Uniform Interface（统一接口）

    场景：
    想象你设计一个大型的微服务架构。每个微服务都有统一的接口规范，这样不同的服务可以相互通信，无需关心实现细节。例如，所有资源都用标准的HTTP方法（GET, POST, PUT, DELETE）操作。

    记忆点：
    统一的标准接口，简化了系统间的交互。

    Scenario:
    Imagine you are designing a large microservices architecture. Each microservice has a unified interface specification, allowing different services to communicate without worrying about implementation details. For example, all resources are manipulated using standard HTTP methods (GET, POST, PUT, DELETE).

    Memory Tip:
    A unified standard interface simplifies interactions between systems.

    4. Resource-Based（资源为基础）

    场景：
    想象你在设计一个电子商务平台，每个商品、订单、用户都是一个资源，每个资源都有唯一的URL标识。客户端通过这些URL来获取或操作资源。

    记忆点：
    资源都有唯一的标识URL，可以通过URL进行操作。

    Scenario:
    Imagine you are designing an e-commerce platform where each product, order, and user is a resource, and each resource has a unique URL identifier. Clients use these URLs to access or manipulate resources.

    Memory Tip:
    Resources have unique URL identifiers and can be operated on via URLs.

    5. Cacheable（可缓存）

    场景：
    想象你在优化一个高流量的网站，利用HTTP缓存机制减轻服务器负担，提高响应速度。通过设置适当的缓存头，客户端可以缓存响应，减少重复请求。

    记忆点：
    合理使用缓存，提高效率和性能。

    Scenario:
    Imagine you are optimizing a high-traffic website by utilizing HTTP caching mechanisms to reduce server load and improve response speed. By setting appropriate cache headers, clients can cache responses, reducing redundant requests.

    Memory Tip:
    Use caching appropriately to improve efficiency and performance.

    6. Layered System（分层系统）

    场景：
    想象你在设计一个复杂的系统，使用分层架构。前端、中间层（如负载均衡器）、后台服务各司其职。中间层可以处理负载均衡、安全、缓存等功能，而客户端只与中间层交互。

    记忆点：
    分层设计，系统更加模块化和可扩展。

    Scenario:
    Imagine you are designing a complex system using a layered architecture. The frontend, middleware (e.g., load balancers), and backend services each have their responsibilities. The middleware handles load balancing, security, caching, etc., and the client interacts only with the middleware.

    Memory Tip:
    Layered design makes the system more modular and scalable.

    7. Code on Demand（按需代码）

    场景：
    想象你在构建一个富客户端应用，服务器根据需要向客户端发送可执行代码（如JavaScript），以增强客户端功能。例如，单页应用程序（SPA）从服务器获取动态内容和脚本。

    记忆点：
    按需加载代码，提升客户端功能和体验。

    Scenario:
    Imagine you are building a rich client application where the server sends executable code (e.g., JavaScript) to the client as needed to enhance client functionality. For example, Single Page Applications (SPAs) fetch dynamic content and scripts from the server.

    Memory Tip:
    Load code on demand to enhance client functionality and experience.

    8. Representation-Oriented（表示为主）

    场景：
    想象你在开发一个API，返回的资源可以有多种表示形式，如JSON、XML、HTML。客户端根据需要请求不同的表示形式（通过Content-Type或Accept头），以便更好地处理数据。

    记忆点：
    资源有多种表示形式，灵活满足不同客户端需求。

    Scenario:
    Imagine you are developing an API that returns resources in various representations, such as JSON, XML, or HTML. Clients request different representations as needed (via Content-Type or Accept headers) to better handle data.

    Memory Tip:
    Resources have multiple representations to flexibly meet different client needs.

4. REST vs. RESTful?
  • REST: A general set of principles for web communication and system design.
  • RESTful: Specialized implementations that adhere to REST principles, specifically for data exchange over HTTP.
