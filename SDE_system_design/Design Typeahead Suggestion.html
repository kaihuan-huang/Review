Design Typeahead Suggestion
  • System requirements
    • Functional:
      • 我将专注于broswer为客户端。我们可以稍后将其扩展到移动设备上。
      • 当用户输入a letter时，Typeahead 服务会找到relevant words。
      • Top priority word 将作为auto complete的候选词。如果此时用户按下 Tab 键，该词将被使用。
      • 一些（default 10，but configurable ）高优先级的词会列在用户正在输入的表单下方，以便用户从中选择一个词。
      • 我假设作为该服务后端一部分的搜索引擎有一种方法可以根据单词的搜索频率来排序。个性化应该被用作这种排序的一部分。
    • Non-Functional:
      • Response time非常重要。在用户输入一个字母后，Typeahead 建议应在 100 毫秒内显示。
      • Scalability: expect 100M DAU.
      • Availability：可以接受不对可用性过于关注。如果 Typeahead 服务短时间不可用，用户 not suffer critical consequences. 。
      • 我假设有一个网络服务可以返回个性化的（通过电子邮件地址）1000 个最可能的搜索词列表。由于这是一个外部网络服务，我们将考虑该服务宕机的错误场景。
      • 另一种错误场景是 Typeahead 服务不可用。我们将确保即使服务器不可用，客户端也能正常操作。
  • Capacity estimation
    • 假设每日活跃用户数（DAU）为1亿，月活跃用户数（MAU）为3亿。
    • 在高峰期，我们考虑每秒2500万个请求。
    • 假设我们为每个用户存储100个最可能的搜索词，另外存储1000个一般最可能的搜索词。
    • 3亿用户 * 100 = 30GB的数据。随着用户数量增加，可能在两年内变成约60GB。
    • 为了扩展性和性能，我认为NoSQL数据库（如MongoDB）是数据存储的一个不错选择。
    • 对于这个数据集，我们不需要复杂的连接操作，因此不需要依赖关系数据库的丰富查询功能。
  • API Design
    • 高性能的关键在于客户端（浏览器）拥有足够的信息来进行 Typeahead 推荐，而不需要每次用户输入时都询问服务器。
    • 客户端将调用以下接口：
    • - `get_words(priority_from, priority_to)`
    • - `get_personalized_words(priority_from, priority_to, user_ID)`
    • 假设流行词按优先级排序，从0到1000。
    • 在页面加载时，客户端会请求最流行的词汇，例如前1000个。这些词汇会针对用户进行个性化处理。
    • 如果客户端有带宽或资源方面的顾虑，它可以选择一次请求较少数量的词汇（例如先请求前100个），然后再请求更多。这类似于分页。
    • 但我预计大多数客户端会请求所有词汇（例如1000个）。如果每个词平均有10个字母，总共约10KB。大多数现代浏览器和网络能够发送和接收这么多数据。
    • 返回的数据（词汇）可以以简单的数组形式传输，使用JSON格式。
  • Database Design
  • High-level Design
    • 页面加载时，客户端调用 get_words() API。
      • CDN 将会非常重要。它caches the most common search work（非个性化）。它能够非常快速地返回常见词（例如在10毫秒的量级内）。
      • 客户端会将候选词存储在一个 Trie（字典树）数据结构中。让客户端在用户输入时quickly narrow down the number of candidates words。
      • 在后台，客户端还会调用 get_personalized_words()。这will hit the Load Balancer directly。这可能需要更多时间（例如在百毫秒甚至秒的量级内）。
      • 词服务器（Words Server）在第一次看到用户时，需要向外部搜索服务请求个性化词列表。一旦收到这些词，它们将存储在数据库中（用于永久permanent存储）和 Redis 中（用于quick访问）。
      • 一旦个性化词汇到达，客户端会将这些词添加到 Trie 数据结构中。
      • 这个 Trie 数据结构可以作为客户端缓存本地存储在客户端中。
  • Request Flow
  • Detailed Component design
    • 一旦构建了 Trie（前缀树），它就能够快速查找可能的单词。例如：
    • a-p-p-l-e
    • -i-c-a-t-i-o-n
      • 这个 Trie 在用户输入第一个字母 “a” 时提供了三种选择：app、apple 和 application。它能迅速排除以其他字母开头的单词。
    • 如果客户端在内存中有这个 Trie 结构，它可以快速遍历并显示单词补全候选项。因此，客户端的任务是维护这个 Trie（构建并存储在客户端存储中）。服务器的任务是通过 CDN 和单词服务器高效地将这些信息提供给客户端。
  • Trade-Offs/ Tech Choices
    • 一个有趣的选择是由谁来合成个性化单词列表和通用单词列表，以创建一个完整的单词列表。
    • 我采用的方法是让客户端接收两个独立的列表并进行合成。
    • 另一种方法是由单词服务器创建合并后的列表并返回给客户端。
    • 我选择第一种方法主要是为了提高响应时间。
    • 通用列表可以非常快速地提供给客户端。因为这是所有用户都使用的相同列表，所以我们可以将其简单地存储在 CDN 中。靠近客户端的 CDN 可以快速返回这个列表。
    • 另一方面，生成个性化列表需要一些时间，特别是当单词服务器需要联系外部搜索服务来获取它时。
    • 因此，最好先将通用列表提供给客户端。个性化列表可以在稍后时间到达。此时，客户端可以将个性化单词添加到 Trie 数据结构中，而用户不会注意到任何变化。
  • Failure Scenarios/Bottlenecks
  • 这种设计在网络丢失时能优雅地处理故障，客户端存储的 Trie 结构保证其功能正常，而浏览器的 Service Worker 可以定期更新候选单词，确保离线时 Trie 依然是最新的。
