  • System requirements
    • Functional:
      • User can
        • Upload
        • delete
        • watch a video( that was previously uploaded)
        • text to search video
    • Non-Functional:
      • Availability
        • Global Reach: Accessible worldwide, always on to retain users.
      • Response Time
        • Low Latency: Watching and searching must have a response time of less than half seconds.
        • No Buffering: Smooth, uninterrupted video playback.
      • Scalability
        • Elastic Growth: Handle increasing users, videos, and views efficiently.
      • Relaxed Consistency: Acceptable for uploads to take some time to become viewable.
      • Device Compatibility: Works well on various devices (different screen sizes, resolutions, computational power, and network bandwidth).
      • Main Error Cases
        • Upload Interruption: Network failure during upload - Implement retries and notify users.
        • Viewing Interruption: Network failure during viewing - Buffering techniques and playback resumption.
  • Capacity estimation
    • 500M DAU
      • 1% of users upload videos.
      • Average user watches 5 videos per day.
      • Peak load - 20% of users access it at the same second. So that's 100M requests per second.
      • Average length of a video is 5 minutes. It would take 50MB of space to store.
        • But there will be videos much longer (e.g. 2 hours).
    • Storage space:
      • Formula: 50MB * 5M users * 1/7 (upload one video per week)
      • Daily Storage: 36 TB
      • Formula: 36TB/day * 365 days/year * 2 years
      • Two-Year Storage: 13 PB
    • estimation
      • Storage Solution:
        • Blob Store: Use a service like Amazon S3 for video storage.
        • Read Heavy: Write once, read many times pattern.
      • Caching and CDNs:
        • Client Proximity: Cache videos close to clients, such as at ISPs and IXPs.
        • Geographic Locations: Use CDNs at major locations to cache videos.
      • Data Replication:
        • Fault Tolerance: Replicate video data in multiple data centers.
        • Read Throughput: Ensure high read throughput with multiple replicas.
  • API design
    • upload_video(user_ID, content, offset, size, tags)
      • This API uploads one chunk of video. The chunk is represented by offset (is it the first chunk, second chunk, ...) and size.
    • start_streaming(user_ID, video_ID, offset, resolution, format, quality)
      • This API downloads initiates video streaming, using a streaming protocol such as DASH or HLS. (standardized.)
        • Depending on the client type and environment it's in, this API lets the client to choose different chunks - by resolution, encoding format, and quality. For example, if the client is a high powered laptop with a good internet connection and is experiencing a high bit rate with the service, it would choose for high resolution and high quality.
    • delete_video(user_ID, video_ID)
    • search_video(user_ID, filters, search text)
      • This API returns a list of video IDs, sorted by how closely it matches the search. It can take filters such as length and language.
    • All APIs return HTTP error code, e.g., 4XX for user errors like unauthorized attempt, or 5XX for system errors. It can also time out (e.g., after waiting for 1 minute).

  • High-level design
  • 其中一个关键组件是比特率的自动调整 automatic adjustment of bit rate。当用户观看视频时，下载块的底层格式the underlying format of the downloaded chunks可能会发生变化：
    • 如果网络带宽 bandwidth增加或减少。
    • 客户端或服务器的资源情况发生变化 resource situation changes。
    • 某些片段segment 可以通过不同的格式format更有效地进行编码 encoded。
    • 为了使 DASH 或 HCL 协议做出此调整，该服务必须提供多个版本video chunks。
    • 为了实现这一点，当 Upload Service 收到原始视频文件时，除了将其分块存储在 Blob Store 中之外，它还会在Message Queue中添加一条消息。
    • Converter Service read info from Message Queue。然后，convert the chunks into多个版本 - 尺寸dimensions、质量、format types, 缩略图thumbnails。它会将所有不同版本存储回 Blob 存储区。
    • 这些不同版本 be cached at CDN and Internet Service Provider cache。
    • 类似地，另一个工作服务 Search Worker 会收到一条消息，并将上传视频的标签提供给 Index Service（基于 Lucene 或 Elastic Search）。Index Service will build a mapping between the tags to the video ID. 。当用户搜索视频时，Search Service 会使用 Index Service find appropriate 视频。
    • 当客户端发送 start_streaming() call，Stream Service 将决定客户端应从where获取数据块。根据此情况，客户 the most appropriate location (via DASH or HCL)下载数据块。ISP Cache 距离客户端最近。但它的缓存可能不如 CDN 的缓存大。因此，观看频率最高的视频应存储在 ISP 中, next most frequently watched videos应存储在 CDN 中。

  • Detailed component design
    • scalable，caching using geographic locations非常重要。原则上，最有可能被客户观看的视频应尽可能靠近客户进行存储。
    • closest location would be a cache in Internet Service Provider. 。其空间有限，most commonly viewed视频（e.g. superbowl highlights and ads）应存储在此处。
    • 更高一级at Internet Exchange Provider.。
    • -> CDN了，公司directly control the size of the cache to store more content.
    • 接下来是数据中心的缓存（例如 Memcached）和 blob 存储。即使在这个级别，geographic preference should be used。例如，最常在日本观看的视频（例如日语视频）应该存储在日本的数据中心。
    • 在多个数据中心复制Replicating video chunks有助于提高可扩展性scalability和容错能力 fault tolerance，以防数据中心goes down for a natural disaster因自然灾害或其他原因而瘫痪。
