  • Design an Efficient Parking Lot System

How would you architect a solution that efficiently manages vehicle entry, exit, and parking allocation? Your design should accommodate different types of vehicles, prioritize convenience for users, and incorporate features to enhance overall functionality and safety.

    • I assume the company:
      • has operations in 10 countries.
      • has 100 parking lots, on average, in one country.
      • On average, each parking lot has a capacity of 200 cars.
        • 20% compact cars

      • 80% of users short term parking: averaging 4hs

      • 10 * 100 = 1000 parking lots

        • User_ID: 20 bytes
        • Car_Type: 1 byte
        • Reservation_start_time: 8 bytes
        • Reservation_end_time: 8 bytes

      • Direction: This estimate leads me to choose a relational database as the storage for reservations. The estimated size is within RDB's limits. RDB also gives strong consistency guarantee, which is suitable for a reservation system because avoiding inconsistency (e.g. double booking) is an important requirement. The response time requirement is not very hard. If reservation takes a couple of seconds (instead of milliseconds), that would be acceptable.

        • 容量估算及数据库选择
        • 通过之前的容量估算，我们确定了每天系统将接收50,000个预订请求，每个预订记录大约为37字节。这使得每天的新增数据量约为1,807KB，2年内累计数据量大约为1,288MB（约1.3GB）。
        • 选择关系型数据库（RDB）
        • 理由
        • 数据量适中：
          • 关系型数据库（如MySQL、PostgreSQL）能够轻松处理每天1,807KB的数据增长，并且总数据量在1.3GB左右。这在关系型数据库的处理能力范围内。
        • 强一致性保证：
          • 关系型数据库提供ACID（原子性、一致性、隔离性、持久性）属性，确保数据一致性。对于预订系统而言，避免重复预订和数据不一致是非常重要的。
        • 结构化数据和复杂查询：
          • 关系型数据库支持复杂的SQL查询，适合处理结构化数据和执行复杂的查询操作。例如，通过JOIN操作，可以轻松地从不同的表中获取数据。
        • 事务处理：
          • 关系型数据库支持事务处理，确保在多步操作中数据的一致性。对于处理预订和支付的系统，这一点尤为重要。
        • 响应时间要求适中：
          • 预订系统的响应时间要求不是特别严格，如果预订过程需要几秒钟而不是毫秒，也是可以接受的。这使得关系型数据库成为一个合适的选择。
        • 实例参考
        • 以下是一些使用关系型数据库处理类似需求的实例参考：
        • MySQL：广泛用于各种Web应用和预订系统，因其高性能和强一致性而著名。
        • PostgreSQL：提供丰富的功能集和强大的SQL支持，被许多企业级应用使用。
        • Oracle Database：提供强大的数据处理能力和一致性保障，适用于大型企业应用。
        • 结论
        • 基于容量估算和系统需求，关系型数据库是存储预订数据的合适选择。它提供了强一致性、结构化数据支持和事务处理能力，确保预订系统的可靠性和数据一致性。此外，关系型数据库能够处理我们估算的数据量，并且在响应时间上满足系统要求。
  • API Design
    • User APIs
      • 1. Check Capacity
        • Endpoint: GET /check_capacity
        • Parameters:
          • lot_ID (String): The ID of the parking lot.
          • vehicle_type (String): The type of vehicle (e.g., standard, compact, electric).
          • start_date_time (Datetime): The start time for the reservation.
          • end_date_time (Datetime): The end time for the reservation.
        • Description: Returns the number of open spots in the given parking lot and the price.
        • Response:
          • available_spots (Integer): The number of open spots.
          • price (Decimal): The price for the specified time period.
      • 2. Reserve Spot
        • Endpoint: POST /reserve_spot
        • Parameters:
          • user_ID (String): The ID of the user.
          • lot_ID (String): The ID of the parking lot.
          • vehicle_type (String): The type of vehicle.
          • start_date_time (Datetime): The start time for the reservation.
          • end_date_time (Datetime): The end time for the reservation.
        • Description: Reserves a parking spot and returns the reservation ID and price.
        • Response:
          • reservation_ID (String): The ID of the reservation.
          • price (Decimal): The price for the reservation.
      • 3. Pay
        • Endpoint: POST /pay
        • Parameters:
          • user_ID (String): The ID of the user.
          • reservation_ID (String): The ID of the reservation.
        • Description: Processes the payment for a reservation.
        • Response:
          • status (String): Payment status (e.g., success, failure).
          • transaction_ID (String): The ID of the payment transaction (if successful).
    • Gate Checking Service APIs
      • 1. Vehicle Arrived
        • Endpoint: POST /vehicle_arrived
        • Parameters:
          • reservation_ID (String): The ID of the reservation.
          • date_time (Datetime): The arrival time of the vehicle.
        • Description: Records the arrival of a vehicle at the parking lot.
        • Response:
          • status (String): Status of the operation (e.g., success, failure).
      • 2. Vehicle Left
        • Endpoint: POST /vehicle_left
        • Parameters:
          • reservation_ID (String): The ID of the reservation.
          • date_time (Datetime): The departure time of the vehicle.
        • Description: Records the departure of a vehicle from the parking lot.
        • Response:
          • status (String): Status of the operation (e.g., success, failure).

  • High-level Design
    • 1. Content Delivery Network (CDN):
      • CDN caches static contents for high performance.
    • 2. Rate Limiter:
      • RL protects the service from Denial of Service (DOS) attacks
    • 3. Load Balancer:
      • distributes requests to App Servers using weighted round robin algorithm.
    • As discussed above, RDB is used as the data store.
      • Relational Database: Serves as the primary data store for reservations, user data, and transaction records.
    • App Server uses Redis Cache for performance gain.
      • by storing frequently accessed data in-memory.
      • Reduces database load and speeds up data retrieval.
  • Request Flow
    • check_capacity() is handled by Reservation Server consulting the data in the database.
    • reserve_spot() is handled by Reservation Server, which creates a new entry in Reservations table.
      • It would create a request for payment, which is queued by Message Queue and submitted to the external Payment System.
      • reserve_spot() may fail if multiple users are trying to book the same spot. Return an error and ask the client to try calling reserve_spot() again.
      • It would also fail if the lot is full. In that case, the client should not retry. Reservation Server may return helpful information such as estimated time spots may open up in the future.
    • vehicle_arrived() and vehicle_left() are handled by Transaction Server, which modifies the Transaction table to keep track of vehicle checkins and checkouts.
      • If vehicle does not arrive after some number of hours (e.g. 24 hours), reservation would be canceled. The user would be charged for 1 day of payment.
      • If inconsistent state happens, e.g., vehicle_left() is called before corresponding vehicle_arrived(), administrator at the lot should be notified. The service would assume the vehicle arrived around the corresponding reservation start time.
  • Detailed Component design
    • 1. Making a Reservation and Payment: Client makes a reservation and pays for it.
    • 2. Vehicle Arrival and Departure: Parking lot system notifies the Transaction System when the user's vehicle arrives and leaves.
    • Algorithm to Find an Open Spot
      • Chop 24 Hours into 15-Minute Slots:
        • One day is divided into 96 slots (24 hours * 4 slots per hour).
        • Each slot is represented by one bit, where 1 indicates occupied and 0 indicates free.
        • Each parking spot requires 96 bits per day, which equals approximately 4KB per year (35040 bits).
      • Reservation Process:
        • Convert start_time and end_time to Index:
          • 00:00 on January 1st is index 0, 00:15 is index 1, and so on.
        • Create Bitmap for Requested Time:
          • Create a bitmap with 1s representing the time between start_time and end_time.
        • Compare Bitmaps:
          • Compare this bitmap with bitmaps representing occupied times of parking spots.
          • Scan from parking spot 0 (closest to entrance) to N (farthest from the entrance).
    • Example Algorithm Steps（This algorithm finds the closest available parking spot for the desired time slot.）
      • Convert Time to Index:
        • Example: Reservation from 10:00 to 11:00.
        • 10:00 -> Index 40 (10 * 4)
        • 11:00 -> Index 44 (11 * 4)
      • Create Bitmap:
        • Reservation Bitmap: 000...0001111000...000 (bits 40 to 44 are 1).
      • Compare Bitmaps:
        • Compare the reservation bitmap to the bitmaps of each parking spot.
        • Find the first parking spot where the AND operation with the reservation bitmap results in zero (000).
    • Optimization
      • Interval Tree: Efficiently manages intervals and supports querying for overlapping intervals.
      • Use Case: Instead of scanning bitmaps, an interval tree can quickly find available spots by querying overlapping intervals.

  • Failure Scenarios/Bottlenecks
    • High Traffic Events:
      • Scenario: Large gatherings near parking lots (e.g., Olympics Games) can stress the reservation workflow (reserve_spot(), pay()).
      • Solution:
        • Horizontal Replication: Replicate software components horizontally to handle increased traffic.
        • Rate Limiting: Implement rate limits to manage traffic spikes and prevent overloading.
        • Database Replication: Ensure database tables are replicated within the data center, across data centers, and in different regions for fault tolerance and disaster recovery.
    • Monitoring and Health Checks:
      • Importance: Monitoring systems are crucial to track the health of all servers and components.
      • Implementation:
