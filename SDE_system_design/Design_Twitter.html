Design Twitter
  • System requirements
    • Functional:
      • User can tweet (send) up to 140 character message.
      • User can follow other users.
      • User can like other users' tweets.
      • User's home feed will show tweets from the users they are following.
      • The home feed will show top K popular tweets, based on the number of likes they receive, and the number of the followers the author has.
        • If I have time, I will add retweeting.
    • Non-Functional:
      • Scalability. It will have to serve a very large population, e.g., 500M DAU.
      • Response time. User has to see tweets quickly. When user opens home feed, the first 10 tweets should show up within 500ms.
      • Availability.
      • Consistency requirement can be a little bit relaxed. It does not require strong consistency like banking transactions. Eventual consistency would suffice. For example, if a user tweets something. If a user in the same geographic region sees the tweet in 1 second, and another user on the other side of the earth sees it after 30 seconds, that would be acceptable.
      • Security, content moderation, and anti abuse protection are all important, but I will not focus on them in this exercise due to lack of time. I will come back to them if I have time.
  • Capacity estimation
    • 500M DAU
      • Each user, on average, tweets twice a day. 1B tweets / day.
      • Each user, on average, views 100 tweets a day.
      • At peak, the system should scale up to 20% of DAU - 100M users - interacting with the system.
        • I will think about storage capacity first.
          • 1B tweets a day. Each tweet is 140 chars. Considering international encoding and metadata, it seems reasonable to assume each message takes up to 512 bytes.
          • So this data increases by 512GB every day. In two years, it would be 365 TB.
          • This goes beyond the typical capacity of RDB. So we'll pick a NoSQL DB.
        • A document based NoSQL DB, e.g., MongoDB, seems like a good candidate. it is more scalable than RDBs. It has schema flexibility, which allows future enhancements of the application.
    • data model:
      • Tweet document:
        • tweet_ID: primary key
        • created_by: user ID
        • posted_time
        • content
        • medialink: link to picture or video content
        • number_of_likes
        • hashtags: list of hashtag strings used in the tweet.
        • users_mentioned: list of users mentioned in the tweet
      • User:
        • user_ID: primary key
        • email
        • name
        • nickname
        • DOB
        • gender
      • Bottleneck analysis:
        • 1 - In this data model, number_of_likes seems to be a challenge to me. If a famous person posts something, and millions of users click "Like" button within one minute, it would overwhelm this document of this database.
          • One approach to overcome this is to break up this into multiple (let's say 100) sub-counters, and make different DB nodes be responsible for each sub-counter. I will come back to this if I have time.
        • 2 - Another important bottleneck is when a user with millions of followers (e.g. famous people) tweets something, the tweet should show up on million of users' home feed. I will look into this deeper in DB design section.
          • They should also receive notifications, but I will punt notifications in this particular exercise.
        • I will consider concurrency in later sections: millions of users viewing the same message concurrently, tweeting messages with the same hash tag, etc.
  • API design
    • tweet(user_ID, content)
      • User represented by user_ID tweets this content.
        • The content will be parsed by the server side for users and hashtags mentioned.
    • follow(follower, followed)
    • User represented by follower user ID will follow user represented by followed user ID.
    • like(user_ID, tweet_ID)
      • User represented by the user_ID likes the tweet represented by tweet_ID.
    • home_feed(user_ID, offset, number)
      • Returns JSON document containing tweets that should be shown on the user's home feed.
      • offset represents where in the list of top tweets it should start from.
      • number represents the number of tweets this API requests.
        • For example, when a user first opens their home feed, it would be from offset 0. It is important for this API to respond quickly, so it would be advisable to keep number relatively small - for instance, 5 or 10 for a mobile app client. After the initial 5 or 10 tweets are shown to the user, the client can request more, potentially in the background, to prepare for more feeds in case user scrolls the home feed.
    • Error handling would follow HTTP error code. For example, 4XX for client side error (e.g. liking a tweet ID that does not exist), 5XX for server side error (e.g. bug causing some of the APIs to fail). It would be important for client to handle too many requests error (429) correctly. For example, if home_feed() API returns 429, it would indicate the servers are too busy to service this potentially expensive call. Therefore, it would be advisable for the client to want for some time (say seconds) before calling this again.
    • Future direction: for a more dynamic content update (e.g. home feed), it might be helpful to consider a WebSocket based API for the client to receive real time update from the service. I will not focus on this for now, but may come back if I have time.
  • Database design
    • MongoDB as the main database for its scalability and schema flexibility.
    • data models:
      • Tweet:
        • tweet_ID: primary key
        • created_by: user ID, index
        • posted_time: index
        • content
        • medialink: link to picture or video content
        • number_of_likes
      • User:
        • user_ID: primary key
        • email: index
        • name
        • nickname
        • DOB
        • gender: index
      • Hashtag:
        • hashtag_ID: primary key
        • hashtag: the hashtag string
        • tweets: tweet_IDs who have this hashtag.
      • UserMentioned:
        • user_ID: the user that is mentioned
        • array of tweet_ID: the tweets that mentions the user
      • Follows:
        • follower: user_ID of the user who is following
        • followed: user_ID of the user being followed
        • timestamp
        • Why Separate?
          • Normalization Concept: Inspired by relational databases, separating the follow information from the User document helps keep the data organized and efficient.
          • Efficient Queries:
            • List Users a User is Following: Quickly find all users that a given user is following.
            • List Followers of a User: Quickly find all users who are following a given user.
      • Hashtags and Tweets:
        • Why Separate? We want to quickly find tweets that use a specific hashtag.
        • How? By keeping hashtags in a separate document, we can easily map hashtags to tweets without having to search through all tweets.
      • User Mentions:
        • Why Separate? A famous user might be mentioned in many tweets quickly.
        • How? Storing mentions in a separate document avoids constantly updating the User document, which would be inefficient.
      • Likes on Tweets:
        • Scalability Issue: When a famous person tweets, millions might like it, creating a bottleneck.
        • Solution: Use a sharded counter service for users with over 1,000 followers.
          • How? Use multiple subcounters in a high-performance store like Redis Cache.
          • Periodic Update: The total likes count is updated in the Tweet document every few minutes.
  • High-level design
    • Components:
      • CDN (Content Delivery Network):
        • Caches static content like images and videos to reduce load on the backend and improve performance.
      • Rate Limiter:
        • Protects the backend servers from being overwhelmed by limiting the number of requests from each client.
      • Load Balancer:
        • Distributes incoming requests across multiple services to ensure even load and high availability.
      • Tweet Service:
        • Handles incoming tweets from clients, stores them in the Tweet database, and creates messages in the Message Queue for further processing.
      • Message Queue:
        • Manages the asynchronous processing of tasks. New tweets generate messages that are pulled by other services.
      • Tweet Analyzer:
        • Pulls messages from the Message Queue, analyzes tweets to extract hashtags and user mentions, and updates the appropriate databases.
      • Content Analyzer:
        • Similar to the Tweet Analyzer, but focuses on content filtering and other content-related processing.
      • Home Feed Service:
        • Constructs and returns a list of recommended tweets for the user in JSON format based on various criteria.
      • Sharded Counter:
        • Manages the counting of likes on tweets, especially for users with many followers, using a sharded counter service and Redis Cache for scalability.
      • DB: MongoDB:
        • The primary database where tweets and other relevant data are stored.
      • Redis Cache:
        • Used by the Sharded Counter for high-performance counting operations.
    • Workflow:
      • Client Interaction:
        • The client sends requests (e.g., posting a tweet, liking a tweet, fetching the home feed) to the system.
        • Static content requests are handled by the CDN for efficiency.
      • Request Handling:
        • Requests pass through the Rate Limiter to prevent abuse and then to the Load Balancer, which distributes the requests to appropriate services.
      • Tweet Processing:
        • New tweets are handled by the Tweet Service, which stores them in MongoDB and adds messages to the Message Queue.
        • The Tweet Analyzer and Content Analyzer pull messages from the queue, process the tweets (e.g., extracting hashtags, content filtering), and update the database.
      • Home Feed Generation:
        • The Home Feed Service constructs a list of recommended tweets for the user and returns it in JSON format.
      • Like Handling:
        • When a tweet is liked, the Sharded Counter service manages the counting using Redis Cache to ensure scalability and periodically updates the Tweet document in MongoDB.
    • Scalability and Fault Tolerance:
      • Data Locality: Data is sharded by user ID and stored in the appropriate datacenter to ensure scalability and reduce latency.
      • Data Replication: Data is replicated across multiple datacenters to ensure fault tolerance and high availability.

  • Detailed component design
    • Home Feed Service Design
      • Purpose: Recommends the top K tweets for a user.
      • Tweet Scoring:
        • Formula:
      • Algorithm:
        • Uses a top-K algorithm based on a heap data structure to ensure the highest score is at the root.
      • Storage:
        • Stores the top K tweets in Redis Cache for fast access.
        • Cache is preferred over a database due to real-time recommendation changes.
        • Redis can persist data to a file if needed.
      • Benefits:
        • Efficient real-time recommendations.
        • Scalable and customizable scoring.
        • Fast access with Redis Cache.
  • Trade offs/Tech choices
    • Database Choice:
      • RDB: Strong consistency but struggles with scalability (estimated 365TB in two years).
      • Wide-Column Database (Cassandra): Better scalability than MongoDB but rigid schema limits feature enhancement.
      • MongoDB: Best balance, offering both scalability and performance, with schema flexibility and normalization advantages.
  • Failure scenarios/bottlenecks
    • Scalability Challenge Scenario 1:
      • Issue: Overwhelming the database if millions of users like a tweet simultaneously.
      • Solution: Use Sharded Counter service with multiple sub-counters to handle likes/dislikes, aggregating counts before writing to the database.
    • Scalability Challenge Scenario 2:
      • Issue: Overwhelming the database when users with millions of followers tweet.
      • Solution: Create a new document that stores recent tweets from famous people. The Home Feed Service pulls information from this document and synthesizes it into the tweets list before returning to the client.
