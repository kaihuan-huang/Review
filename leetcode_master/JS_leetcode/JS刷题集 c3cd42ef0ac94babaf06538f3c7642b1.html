1. What's the output?
function sayHi() {
  console.log(name);
  console.log(age);
  var name = 'Lydia';
  let age = 21;
}

sayHi();
  • A: Lydia and undefined
  • B: Lydia and ReferenceError
  • C: ReferenceError and 21
  • D: undefined and ReferenceError
  • Answer

    Answer: D

    • var声明的变量会被提升Hoisting，并且在初始化之前默认值为undefined。
    • let const 声明的变量也会被提升Hoisting，但在初始化之前访问它们会导致ReferenceError，这是由于暂时性死区Temporal Dead Zone, TDZ的存在。

    Within the function, we first declare the name variable with the var keyword. This means that the variable gets hoisted (memory space is set up during the creation phase) with the default value of undefined, until we actually get to the line where we define the variable. We haven't defined the variable yet on the line where we try to log the name variable, so it still holds the value of undefined.

    Variables with the let keyword (and const) are hoisted, but unlike var, don't get initialized. They are not accessible before the line we declare (initialize) them. This is called the "temporal dead zone". When we try to access the variables before they are declared, JavaScript throws a ReferenceError.

30. the event loop in JavaScript, where asynchronous callbacks (even with a delay of 0 milliseconds) are executed after the current stack of synchronous code.
const foo = () => console.log('First');
const bar = () => setTimeout(() => console.log('Second'));
const baz = () => console.log('Third');

bar();
foo();
baz();
  • First Third Second

    We have a setTimeout function and invoked it first. Yet, it was logged last.

    This is because in browsers, we don't just have the runtime engine, we also have something called a WebAPI. The WebAPI gives us the setTimeout function to start with, and for example the DOM.

    After the callback is pushed to the WebAPI, the setTimeout function itself (but not the callback!) is popped off the stack.

    Now, foo gets invoked, and "First" is being logged.

    foo is popped off the stack, and baz gets invoked. "Third" gets logged.

    The WebAPI can't just add stuff to the stack whenever it's ready. Instead, it pushes the callback function to something called the queue.

    This is where an event loop starts to work. An event loop looks at the stack and task queue. If the stack is empty, it takes the first thing on the queue and pushes it onto the stack.

    bar gets invoked, "Second" gets logged, and it's popped off the stack.
